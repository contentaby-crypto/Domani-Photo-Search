# TELEGRAM_INTEGRATION_SPEC

## 1. Назначение

Документ описывает интеграцию MVP с Telegram Bot API: входящие сообщения, webhook, inline-кнопки, формат пользовательского ответа и правила отправки медиа.

## 2. Общая схема

Telegram используется как **клиентский интерфейс**.
Бот не хранит поисковую логику внутри Telegram-обработчиков. Он только:
- принимает апдейты;
- валидирует вход;
- вызывает внутренние сервисы;
- отображает результат пользователю.

## 3. Способ интеграции

Для MVP использовать **webhook**.

### Причины
- предсказуемое поведение в проде;
- проще деплой в web environment;
- быстрее реакция на входящие события;
- лучше подходит для hosted backend.

### Требования
- HTTPS endpoint;
- валидация secret token;
- отдельный маршрут, например `/webhook/telegram`.

## 4. Входящие типы событий

### Поддерживаемые типы
- текстовое сообщение пользователя;
- callback query от inline-кнопок;
- команда `/start`;
- команда `/help`;
- команда `/find` (опционально).

### Не поддерживаются на MVP
- голосовые сообщения;
- фото как запрос;
- документы;
- inline mode;
- группы и каналы.

## 5. Команды

### `/start`
Показывает:
- краткое описание бота;
- 2–3 примера запросов;
- предупреждение, что лучше задавать предметный запрос.

### `/help`
Показывает:
- что умеет бот;
- примеры корректных запросов;
- как уточнять поиск.

### `/find`
Опциональный алиас для явного режима поиска.
Если после команды есть текст — обрабатывается как запрос.
Если нет — бот просит ввести текст.

## 6. Основной пользовательский сценарий

### 6.1 Узкий запрос
Пользователь: `Найди серую кухню у Игнатьева`

Бот:
1. вызывает search service;
2. вызывает ranking service;
3. отправляет 1–10 карточек.

### 6.2 Широкий запрос
Пользователь: `Покажи кухни`

Бот:
1. получает `delivery_mode = ask_user`;
2. отправляет сообщение:
   `Найдено N фото. Уточнить запрос или прислать всё?`
3. показывает 2 inline-кнопки.

### 6.3 Пустой результат
Пользователь: `квартира марсианина`

Бот:
1. получает `delivery_mode = not_found`;
2. отправляет сообщение:
   `Совпадений не найдено. Попробуйте уточнить объект, комнату или стиль.`

## 7. Inline-кнопки

### Кнопки при широком результате
- `Прислать найденное`
- `Уточнить запрос`

### Формат callback_data

Рекомендуемый формат:
```text
action:request_id:page
```

Примеры:
- `send_all:b6d1c769-0b73-4c91-b6c9-7a1eb4438d11:1`
- `refine:b6d1c769-0b73-4c91-b6c9-7a1eb4438d11:0`

### Ограничения
- `callback_data` должно быть коротким;
- нельзя сериализовать туда весь shortlist;
- подробности должны храниться на сервере по `request_id`.

## 8. Формат сообщений

### 8.1 Текстовый ответ при ask_user
```text
Найдено 18 фото. Уточнить запрос или прислать всё?
```

### 8.2 Карточка фото
Каждое фото должно сопровождаться минимум:
- превью;
- названием файла;
- объектом;
- ключевыми тегами;
- ссылкой или ID оригинала.

### 8.3 Сообщение об ошибке
```text
Не получилось обработать запрос. Попробуйте еще раз позже.
```

## 9. Отправка медиа

### Базовое правило
Если найдено несколько карточек, бот отправляет их **батчами**.

### Рекомендуемый размер батча
- 5 карточек за раз в MVP;
- конфигурируется через env.

### Возможные режимы отправки
1. `sendPhoto` с caption на каждую карточку;
2. `sendMediaGroup` для компактной пачки.

### MVP-рекомендация
Использовать:
- `sendPhoto` для top 1–10;
- `sendMediaGroup` или батчевую отправку при `send_all`.

## 10. Состояние диалога

На стороне backend нужно хранить короткоживущий session state:
- `session_id`
- `last_request_id`
- `last_query_text`
- `last_normalized_query`
- `last_delivery_mode`
- `last_shortlist_ids`

Это нужно для сценариев:
- refine;
- send_all;
- повторные вопросы в рамках текущего поиска.

## 11. Ограничения Telegram-слоя

Bot layer не должен:
- принимать решение о релевантности фото;
- сам интерпретировать словари;
- хранить полный shortlist в callback_data;
- обращаться к OpenAI, минуя ranking service.

## 12. Обработка ошибок

### invalid_query
Бот просит сократить или уточнить запрос.

### no_matches
Бот показывает подсказки по переформулировке.

### ranking_mismatch
Бот игнорирует ranking-ответ и показывает deterministic results.

### internal_error
Бот отдает стандартное сообщение и логирует `request_id`.

## 13. Безопасность

- webhook только по HTTPS;
- проверка заголовка secret token;
- token бота хранится только в env;
- логировать тело Telegram update без чувствительных данных и без переполнения логов.

## 14. Контракт с bot-handlers

### Вход handler'а
- Telegram update
- конфиг
- зависимости: search client, ranking client, session store

### Выход
- отправка сообщений в Telegram;
- лог события;
- обновление session state.

## 15. Тестовые сценарии Telegram-интеграции

1. `/start` отдает onboarding.
2. Текстовый запрос уходит в search.
3. При `ask_user` показываются 2 кнопки.
4. `send_all` отправляет все карточки батчами без дублей.
5. `refine` не теряет контекст последнего запроса.
6. Ошибка ranking не ломает ответ пользователю.
