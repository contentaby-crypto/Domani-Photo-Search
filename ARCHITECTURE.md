# ARCHITECTURE

## 1. Назначение

Этот документ фиксирует инженерную архитектуру MVP проекта **поисковой системы по базе фотографий Domani**.
Архитектура рассчитана на реализацию через Codex и на поддержку трех основных контуров:

1. **Telegram Bot Layer**
2. **Search Service**
3. **LLM Ranking Service**

Ключевой принцип MVP: **детерминированный shortlist → LLM только для финального ранжирования**.

## 2. Архитектурные цели

- обеспечить предсказуемый и воспроизводимый поиск;
- исключить поиск по всей базе напрямую через LLM;
- отделить Telegram-канал от поисковой логики;
- обеспечить аудит и трассируемость каждого запроса;
- сделать структуру кода понятной для Codex и команды.

## 3. Логические компоненты

### 3.1 Telegram Bot Layer

**Ответственность:**
- прием входящих сообщений и callback-кнопок из Telegram;
- валидация пользовательского ввода;
- вызов Search Service;
- вызов Ranking Service;
- выбор сценария ответа пользователю;
- пакетная отправка превью и карточек.

**Не делает:**
- не нормализует запрос сам;
- не считает shortlist;
- не вызывает OpenAI напрямую для поиска по всей базе.

### 3.2 Search Service

**Ответственность:**
- загрузка и валидация индексированной базы;
- нормализация свободного текста;
- выделение сущностей по словарям;
- применение hard filters;
- deterministic scoring;
- построение shortlist (обычно 40–50 карточек);
- возврат trace поиска.

**Основной контракт:**
- вход: `query_text`, `context`, `top_k`;
- выход: `normalized_query`, `shortlist`, `delivery_mode`, `trace`.

### 3.3 LLM Ranking Service

**Ответственность:**
- принять исходный запрос и shortlist;
- отсортировать shortlist;
- выбрать top N (обычно 5–10 карточек);
- вернуть краткое объяснение релевантности.

**Жесткие ограничения:**
- LLM получает только shortlist;
- LLM не создает новые `photo_id`;
- LLM не придумывает новые URL, объекты и теги;
- при mismatch deterministic shortlist остается authoritative.

### 3.4 Data Layer

**Состав:**
- исходный CSV;
- нормализованные словари;
- индексированный dataset;
- артефакты reindex;
- эталонный golden dataset для QA.

**MVP-решение:**
на этапе MVP допустимо файловое хранение:
- `data/raw/*.csv`
- `data/normalized/*.json`
- `data/index/*.jsonl|parquet`

### 3.5 Logging & Audit Layer

**Ответственность:**
- единый `request_id`;
- логирование каждого шага пайплайна;
- хранение `normalized_query`, `shortlist_ids`, `ranked_ids`, ошибок;
- сбор метрик качества поиска.

## 4. Runtime-поток запроса

### 4.1 Основной happy path

1. Пользователь отправляет текст в Telegram.
2. Bot Layer валидирует длину и тип сообщения.
3. Bot Layer вызывает `POST /v1/search/query`.
4. Search Service:
   - нормализует запрос,
   - применяет словари,
   - строит shortlist,
   - возвращает `delivery_mode`.
5. Если `delivery_mode = direct`:
   - Bot Layer вызывает `POST /v1/ranking/rank`,
   - получает top N,
   - отправляет пользователю результат.
6. Если `delivery_mode = ask_user`:
   - Bot Layer показывает количество найденных фото и inline-кнопки.
7. Если пользователь нажимает `send_all`:
   - Bot Layer отправляет карточки батчами.
8. Если пользователь нажимает `refine`:
   - Bot Layer показывает подсказки и сохраняет контекст.

## 5. Deployment view

Для MVP рекомендуется один backend-процесс или один монорепозиторий с тремя логическими модулями:

- `app/bot/`
- `app/search/`
- `app/ranking/`

Допустимы два режима запуска:

### Вариант A. Monolith MVP
Один Python-сервис, внутри которого есть:
- webhook handler,
- search module,
- ranking module.

**Плюсы:** быстрее старт, проще локальная разработка.
**Минусы:** слабее границы между слоями.

### Вариант B. Modular monolith (рекомендуется)
Один deployable service, но отдельные внутренние пакеты и строгие интерфейсы.
Это лучший вариант для Codex и MVP.

## 6. Технологический стек MVP

### Базовый стек
- Python 3.12
- FastAPI — HTTP API и webhook endpoint
- aiogram — Telegram Bot API integration
- Pydantic v2 — схемы данных и контрактов
- OpenAI Python SDK — вызов Ranking Service через Responses API
- pandas / pyarrow — загрузка и индексация CSV
- rapidfuzz — alias / fuzzy matching
- pymorphy3 — морфологическая нормализация русского языка
- structlog или стандартный `logging` в JSON-формате
- pytest — тесты

## 7. Нефункциональные решения

### Воспроизводимость
Детерминированный shortlist должен быть одинаковым для одной и той же версии данных и словарей.

### Наблюдаемость
Каждый шаг логируется с `request_id`, `session_id`, `duration_ms`.

### Безопасность
- все внешние ключи только в env;
- webhook защищен secret token;
- OpenAI API вызывается только сервером.

### Масштабируемость
На MVP индекс может храниться на диске; при росте можно вынести data/index в отдельное хранилище без смены контрактов.

## 8. Границы ответственности между слоями

| Слой | Что делает | Что не делает |
|---|---|---|
| Bot | принимает/отправляет Telegram-сообщения | не ищет по базе |
| Search | нормализация, shortlist, trace | не отправляет сообщения |
| Ranking | сортирует shortlist через LLM | не строит shortlist |
| Data | хранит базу, словари, индекс | не знает о Telegram |
| Logging | аудит и диагностика | не влияет на бизнес-логику |

## 9. Принципы проектирования для Codex

1. Один модуль = одна зона ответственности.
2. Все контракты описаны Pydantic-моделями.
3. Все конфигурации выносятся в `config.py` + env.
4. Все словари и prompt templates лежат в отдельных файлах.
5. Все побочные эффекты (Telegram/OpenAI/file I/O) инкапсулируются в adapters/clients.
6. Бизнес-правила поиска не смешиваются с транспортом.
7. Любой шаг должен быть воспроизводимым локально через CLI или тест.

## 10. Решения MVP, принятые заранее

- Внутренний стабильный ключ карточки: `photo_id`.
- Источник истины по поиску: индекс + словари.
- LLM используется только после deterministic shortlist.
- Telegram-бот не обращается к OpenAI напрямую по всей базе.
- При конфликте результата ranking и shortlist authoritative остается shortlist.
