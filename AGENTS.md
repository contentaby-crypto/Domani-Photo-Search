# AGENTS.md

Правила для Codex и всех разработчиков, которые вносят изменения в репозиторий.

## 1. Общие принципы

1. Не менять продуктовую логику без явного изменения соответствующей спецификации.
2. Сначала обновлять код, затем тесты, затем документацию, если поведение изменилось.
3. Не встраивать бизнес-правила напрямую в Telegram-слой, если они относятся к поиску или ранжированию.
4. Все изменения должны быть минимальными, локальными и обратимо проверяемыми.

## 2. Что считается source of truth

- MVP-логика поиска: `ARCHITECTURE.md`
- словари и правила нормализации: `01_...docx` и `DATA_MODEL.md`
- внутренний API-контракт: `02_...docx` и `src/domani_photo_search/api`
- тестовая стратегия: `TEST_PLAN.md`
- формат данных: `DATA_MODEL.md`

Если код противоречит документам, сначала обновляется документ, потом код.

## 3. Границы ответственности модулей

- `bot/` — только Telegram update handling, callbacks, delivery.
- `search/` — нормализация, фильтры, scoring, shortlist.
- `llm/` — формирование ranking input/output, валидация ответа модели.
- `indexing/` — ingest CSV, нормализация тегов, построение индекса.
- `schemas/` — Pydantic-модели контрактов.
- `services/` — orchestration между search, ranking и delivery.

## 4. Что Codex не должен делать

- Не придумывать новые бизнес-сущности без документации.
- Не менять веса ранжирования молча.
- Не обращаться к LLM по всей базе.
- Не добавлять неиспользуемые зависимости.
- Не хранить секреты в коде.
- Не смешивать sync и async без необходимости.

## 5. Правила по изменениям

### Разрешено без согласования

- исправления багов;
- покрытие тестами;
- рефакторинг без изменения контрактов;
- улучшение логирования;
- типизация и валидация.

### Требует обновления спецификаций

- новые endpoint'ы;
- изменение формата callback_data;
- изменение схемы `photo_card`;
- изменение scoring weights;
- изменение env-переменных;
- изменение процесса индексации.

## 6. Стиль кода

- Python 3.12
- Полная type annotation для публичных функций.
- Pydantic для внешних контрактов.
- Логирование через structured logging.
- Модули должны быть небольшими и тематическими.
- Нельзя писать функции на 200+ строк без крайней необходимости.

## 7. Тестирование при любом изменении

Минимум:

```bash
make format
make lint
make test
```

Если изменялся контракт или поиск, дополнительно:

```bash
make test-contract
make test-integration
```

## 8. Правила для prompt/inference слоя

- Модель: `chatgpt-5.4`
- В модель передается только shortlist.
- На выходе допускается только перестановка или сужение shortlist.
- Ответ модели должен проходить post-validation.
- Если модель вернула `photo_id` вне shortlist, результат LLM игнорируется.

## 9. Правила по данным

- CSV из `data/raw/` не изменяется вручную внутри runtime-кода.
- Все преобразования выполняются в `indexing/`.
- Словари в `config/dictionaries/` версионируются вместе с кодом.
- `photo_id` после публикации нельзя менять.

## 10. Формат PR

Каждый PR должен отвечать на 4 вопроса:

1. Что изменено?
2. Почему это нужно?
3. Какие тесты добавлены/обновлены?
4. Какие документы нужно считать обновленными?
